Weeks 9-12

Week 9

Week 9 consisted of testing all the scripts on the new server. A few errors were spotted where users were not being created properly. Minor modifications to a line in the script fixed this and another test run later that day was successful.

The second script, to manage the MySQL side of the server (creation of users, permissions set on databases) was then tested. This was mostly successful but Rob came to us after looking through PhpmyAdmin and discovering a few permission errors on certain databases. An additional GRANT statement was added to the script and this issue was resolved. Going back to Rob and asking him to look through PhpmyAdmin again, he was happy the issue had been rectified and all users had their correct permissions including root or admin users.

Week 10

By week 10, the DB2 server was nearing completion with just minor house-keeping needed, and documentation to be completed. Since Jordan had done a lot of the smaller setups and configuration, I let him undertake the task of completing documentation and a set up guide.

I had now finished focusing on DB2, other than demonstrations and to help Jordan with any questions.

I decided to turn my attention to the DB3 course. Rob always said DB3 never needed to be looked at this semester, but it was always going to be nice if it did get looked into. He was happy if 2021 Semester 2 students started looking into it as it wasn't neccessary for this year, but given DB2 was near completion, I decided to start working on this.

First task began with refreshing myself on how DB3 was run. Going through labs and reading through my previously created DB3 load script, I remembered DB3 ran on 2 servers. A Ubuntu Docker server where students would launch MSSQL Docker containers, and a Windows Server server where students would perform various DBA tasks. Talking with Rob, we needed to decide on the appropriate solution, would we look into a similar 'lift and shift' that we did with DB2, or could we explore hosted services options? Going through what students need to do and what they need to achieve, we couldn't find a simple hosted service solution, and decided for simplicity, ease of use and not having to spend the time to further educate lecturers on how to use new software, a lift and shift was the way to go.

Week 11

Next step after we decided on a lift and shift option, was to go into the DB3 servers and see what type of hardware these servers are running, what software, and check if there is any current resource issues. If we are merging into Azure servers, it makes sense to explore the options of upgrades, both with hardware and software.

I started looking into the DB3 Docker server first as this was the most accessible and I also like to try and further my knowledge and experience with Docker and containers. The DB3 Docker server runs 2 virtual CPU's, 16GB RAM, and an 80GB HDD. Looking into the CPU's more, these were expensive server CPU's running 12 cores each @ 2.2Ghz. A concern for me was going to be the cost of running a machine, similarly or better specc'd, in the cloud. DB2 was a relatively low cost course to set up, and running this single Docker server was going to be more than the cost to run DB2 alone.

I had a conversation with Rob and expressed my concern about the cost involved. I feel like I have a duty to optimise costs as much as possible for Otago Polytechnic because as a business, you want to minimise outgoing or ongoing costs, but at the same time we want to deliver a high quality of service and teaching. I gave Rob a list of servers similar to what we were currently running and approximate costs. To my surprise, Rob assured me that all of these were within budget for the Polytechnic and I could continue exploring options.

Week 12

During week 12 I was exploring Azure VM options. DB3 Docker is a very memory intensive server and I needed to look into VMs built for a specific purpose. This was all new to me as DB2 was able to be run on a general purpose VM.

I looked into VMs and was overwhelmed by all the different series, purposed and the great variance in price. I got in touch with both Rob and Faisal to help point me in the direction of learning which VMs were going to be suitable and to learn about each series of VMs

Faisal pointed me in the direction of an Azure resource which went through each series. After looking into it, a B series VM was looking to be suitable. The B stands for burstable. This was talking about CPU performance. A burstable VM's CPU will run at a baseline performance percentage but during high loads, that percentage can burst offering higher performance during high loads. This burst varies for each machine, but after consulting with Rob, a B2mS server looked like a good option to explore. That particular VM runs at 60% but can burst to 200%.

A burstable VM looked like a good option as high loads would only be expected during class times where students are launching and working with containers. Assuming 2 streams of classes with 4 hours class time each per week, the VM would run at its full capacity for 8 hours per week, but allowing some additional time for students working outside of classroom hours.
